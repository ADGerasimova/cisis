# SQL Migrations — CISIS

## Как это работает

В этом проекте миграции базы данных управляются вручную через SQL-файлы, 
а не через Django `makemigrations`/`migrate`. Все модели Django используют 
`managed = False`.

## Структура

```
sql_migrations/
├── README.md                          ← этот файл
├── full_schema.sql                    ← полная схема БД (для развёртывания с нуля)
└── incremental/                       ← инкрементальные миграции
    ├── 001_v3_14_0_audit_log.sql      ← первая пронумерованная миграция
    └── ...                            ← будущие миграции
```

## Развёртывание с нуля (новый сервер / новый разработчик)

1. Создать базу данных:
   ```sql
   CREATE DATABASE "CISIS";
   ```

2. Выполнить полную схему:
   ```
   pgAdmin → Query Tool → открыть full_schema.sql → Execute
   ```

3. Выполнить все инкрементальные миграции по порядку номеров:
   ```
   001_v3_14_0_audit_log.sql
   002_...
   ```

4. Запустить Django-миграции (для системных таблиц):
   ```bash
   python manage.py migrate
   ```

5. Заполнить справочные данные (laboratories, standards и т.д.)

## Обновление существующей базы

Выполнять только **новые** инкрементальные миграции, которых ещё нет в вашей базе.
Номер миграции указывает порядок применения.

## Правила

- **full_schema.sql** обновляется после каждого изменения структуры БД
  (пересоздаётся через `pg_dump --schema-only`)
- **Инкрементальные миграции** нумеруются последовательно: 001, 002, 003...
- Каждая миграция обёрнута в `BEGIN; ... COMMIT;` для атомарности
- Файлы миграций **не удаляются** после применения — это история
